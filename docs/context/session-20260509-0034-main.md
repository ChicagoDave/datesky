### Investigation
- Traced the pollution path end-to-end: `TagInput.tsx` (client sanitization was only `trim().toLowerCase().replace(/^#/, "")` — no character whitelist), `src/app/api/profile/route.ts` (PUT handler had zero tag validation), `lexicons/app/nomare/profile.json` (only constraint was `maxGraphemes: 64` — too permissive), display path (`ProfileCard.tsx`, `ProfileView.tsx` — React escaping blocked XSS but pollution still degraded matching), and `src/lib/db/queries.ts:119` (`tag = ?` lookups silently fail to match URL-shaped tags against legitimate ones).

### Tag validation module (new single source of truth)
- `src/lib/profile/tag-validation.ts` — pure function `isValidProfileTag(value: string): { ok: true } | { ok: false; reason: string }`. Enforces `^[a-z0-9][a-z0-9-]{0,62}[a-z0-9]$`: lowercase alphanumeric + internal hyphens, 2–64 characters. Rejects URLs, whitespace, dots, slashes, colons, uppercase, underscores, emoji. Module header documents the rule as a locked invariant with the user-approved tradeoff (rejects `node.js`, `c++` — tighter rule preferred).

### Tag cleanup module (new)
- `src/lib/db/tag-cleanup.ts` — `removeInvalidTags(db: Database.Database, opts: { apply: boolean }): CleanupResult`. Single-transaction delete pass over the `profile_tags` table. Returns `{ scanned, invalid, removed, apply }`. Caller owns DB lifecycle. Header documents the durability caveat: Jetstream re-sync can re-pollute the index if `upsertProfile` does not filter tags at ingest time — the open item for the next session.

### Client-side enforcement
- `src/components/TagInput.tsx` — `addTag` now calls `isValidProfileTag` before inserting. On rejection, sets a state-managed `error: string | null` and renders an inline red `role="alert"` paragraph under the input. Error clears on the next keystroke. Existing duplicate-tag check and max-tags guard preserved unchanged.

### Server-side enforcement
- `src/app/api/profile/route.ts` — PUT handler imports the validator. Before calling `putRecord`, iterates `record.tags` (when array) and returns `400 { error: "Invalid tag(s)", invalid: [{ tag, reason }, ...] }` if any fail. A code comment frames this as defense in depth against direct API callers and future Nomare clients — the client guard is not the only gate.

### Operator tooling
- `scripts/scan-invalid-tags.ts` — read-only scan. Opens DB read-only. Iterates every row in `profile_tags`, applies `isValidProfileTag`, and prints `DID | tag | reason` for each failure. Zero mutations.
- `scripts/cleanup-invalid-tags.ts` — operator CLI. Dry-run by default. `--apply` flag mutates. In dry-run mode, opens DB read-only. Prints per-row lines (`[dry-run] would remove ...` or `removed ...`) and a summary footer. Footer in dry-run mode instructs the operator to re-run with `--apply`. Prints a caveat that PDS records are not touched — the affected user must re-save their profile via the validated form to update their AT Protocol record.

## Key Decisions

### 1. Tag shape rule: STRICT — locked by user
`^[a-z0-9][a-z0-9-]{0,62}[a-z0-9]$` — lowercase alphanumeric + internal hyphens, 2–64 characters. The user explicitly chose this over a permissive rule that would accept dots or underscores. Documented tradeoff: rejects `node.js` and `c++`. This rule is now the invariant in `src/lib/profile/tag-validation.ts` and is shared verbatim by all enforcement points.

### 2. Single source of truth for the validator (CLAUDE.md rule 7b)
The validator lives exclusively at `src/lib/profile/tag-validation.ts` and is imported by client (`TagInput.tsx`), server (`route.ts`), the scan script, and the cleanup script. No re-declaration or duplication. Boundary Statement was produced in the conversation before editing: OWNER = Profile bounded context, SHARED = no (invariant is the same for all consumers), PROMISE = narrowing the lexicon constraint (not weakening it), ALTERNATIVES = inline in route or in TagInput — rejected because both allow drift between the two enforcement points.

### 3. Cleanup is dry-run by default
The `--apply` flag must be passed explicitly to mutate. Without it, the script opens the DB read-only and cannot accidentally write. This matches the pattern established by the rebrand scripts in the prior session.

### 4. PDS records are out of scope for cleanup
Only the local SQLite `profile_tags` index is cleaned. The AT Protocol record on the user's PDS can only be updated by the user's own OAuth session. The cleanup script prints this caveat explicitly. The affected user must re-save their profile via the now-validated form.
