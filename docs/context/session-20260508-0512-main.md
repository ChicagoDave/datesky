## Completed

### Phase 3 Deploy Followup — Private Key Rotation

Pre-session journal showed `invalid_client_metadata: client_id does not match` failures from `bsky.social/oauth/par`. Root cause: Phase 3 committed `kid: nomare-key-1` as the public JWK and updated `src/lib/atproto/oauth-client.ts`, but production `.env`'s `OAUTH_PRIVATE_KEY` was still the old `datesky-key-1` private — the service was advertising one keypair and signing with another.

Operator had previously transferred the `nomare-key-1` private JWK to `~/.nomare-private-key.json.tmp` (mode 0600) on plover.net. Public `x`/`y`/`kid` alignment between `public/jwks.json` and that file was verified without reading the `d` field into LLM context. An atomic Python rewrite of the `OAUTH_PRIVATE_KEY` line in `.env` updated the value while preserving 0o664 perms. Operator restarted `datesky.service`; both local `:3003/jwks.json` and public `datesky.app/jwks.json` confirmed the new public JWK being served.

### Stale `.next/` Build Identified and Rebuilt

User reported the live site still displaying "DateSky" branding despite Phase 1–3 source changes. Diagnosis: `.next/` was built 2026-05-06, predating all three rebrand phases. Since `next start` reads from `.next/` at runtime, the production service was rendering the old strings. `npm run build` rebuilt the artifacts (result: 0 "DateSky", 92 "Nomare"). Operator restarted the service.

### Phase 4 Step A — Apache Vhosts for nomare.net

Wrote two new Apache configuration templates:

- `scripts/apache/nomare.net.conf` — HTTP VirtualHost that issues a 301 to HTTPS
- `scripts/apache/nomare.net-le-ssl.conf` — HTTPS VirtualHost with Let's Encrypt cert paths and reverse proxy to `:3003`

Both are modeled on the existing `datesky.app` vhosts already committed to `scripts/apache/`. Operator executed the standard Apache sequence: `cp` to `/etc/apache2/sites-available/`, `a2ensite nomare.net.conf`, `certbot certonly --apache -d nomare.net`, `a2ensite nomare.net-le-ssl.conf`, `systemctl reload apache2`. Verified: HTTPS 200 at `nomare.net/`, `/jwks.json` returns `kid: nomare-key-1` with matching `x`/`y`, `/client-metadata.json` is self-consistent, HTTP→HTTPS 301 working, Let's Encrypt cert valid through 2026-08-06.

### `PUBLIC_URL` Mismatch Diagnosed and Fixed

OAuth login still 500'd after the vhost came up. Journal showed the same `invalid_client_metadata: client_id does not match` error. Root cause: production `.env` had `PUBLIC_URL=https://datesky.app`, so the OAuth client claimed `client_id = https://datesky.app/client-metadata.json` in the PAR, while the document at that URL (Phase 3 flipped) advertises `client_id = https://nomare.net/...` — a mismatch. Atomic Python rewrite of the `PUBLIC_URL` line in `.env` to `https://nomare.net`. This also surfaced as a drift between the updated `.env.example` (Phase 3 changed it) and the stale production `.env` that was never refreshed.
