## Key Decisions

### 1. Refactor Jetstream to use queries.ts (single source of truth)
User chose to delete Jetstream's duplicated prepared-statement upsert logic and import from `queries.ts` rather than add the filter inline. Rationale: two separate upsert paths would mean two places to maintain the filter rule, reintroducing the same drift risk that motivated the validator module last session.

### 2. Filter both browse and match
Consistent visibility lever — if a profile is hidden from browse it is hidden from match too. User chose this over filtering only browse because a polluted profile reaching a match recommendation would undermine trust in the same way as appearing in browse.

### 3. Site-wide banner placement
User chose the top-of-layout banner over an editor-only warning so polluted users cannot navigate away without seeing the notice. The banner names the specific offending tags and links directly to the profile editor.

### 4. Hard-block visibility, soft-restore on resave
The `has_invalid_tags` flag is set and cleared automatically by `upsertProfile`. No separate fix-up UI flow is needed; users re-save through the existing validated form and the flag clears in the same transaction. This keeps the restore path zero-friction.

### 5. Backfill via operator script, not auto-run on app start
The operator owns the timing and dry-run preview before applying. Auto-running on start would hide profiles immediately on deploy without giving the operator a chance to audit, and could mask a bug in the flag-computation logic.
