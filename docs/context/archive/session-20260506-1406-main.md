## Key Decisions

### 1. Top sticky header + segmented nav replaces bottom tab bar

The iOS Safari bottom tab bar wobble is caused by the browser's dynamic URL-bar resize colliding with a fixed `bottom: 0` element. A top sticky header with in-page segmented navigation avoids the collision entirely. Tap targets move well above the URL-bar zone. This also matches the dominant pattern on web-first products while remaining comfortable on mobile.

### 2. ZIP-based community-tier discovery model

The profile edit gains a private ZIP code field (never shown publicly). Proximity is calculated server-side and expressed as three tiers: Community (same ZIP), Extended (adjacent ZIP codes), Distant (elsewhere). This replaces the broken free-text location search and gives the browse feed a meaningful social-proximity signal borrowed from thefold's latent-demand concept.

### 3. Trifecta intentions — Dating, Friends, Community as peer modes

Rather than treating Dating as the primary intent and Friends/Community as secondary filters, all three are first-class toggleable intentions. A user may have any combination active. This opens the product to users who are not seeking dates, broadens retention, and supports the pavilion/community metaphor without requiring separate apps or profiles.

### 4. Opt-in adaptive theming decoupled from identity

The theming system is personalization, not identity assertion. A user who selects the Trans palette is not required to have filled in a gender identity field; a user who identifies as bisexual is not defaulted to the bisexual palette. The two systems are independent. This avoids the dual failure modes of (a) forcing users to out themselves through theme choice and (b) making assumptions about preference from identity data.

The default Dusk palette (warm coral / lavender / sage) is intentionally neutral — it reads as "modern and warm" without reading as a specific community flag. This means a straight user's first experience is not visually coded as queer-first, addressing the inclusion-by-design goal without compromising the app's core affirmation of all identities.

### 5. Cross-theme profile detail rule

When viewing another person's profile, your shell (nav, buttons, chrome) stays in your theme; the hero region of the viewed profile picks up that person's accent colors. Action buttons (message, like) remain in your theme for affordance consistency — a button should look the same everywhere so users learn it. This gives each profile a personal visual identity without disorienting the viewer's action affordances.
