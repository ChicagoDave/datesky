## Completed

### 1. Home Page Auth-Aware CTA

`src/app/page.tsx` was a client component that rendered `<LoginButton />` unconditionally. The Nav correctly showed `@handle` for signed-in users (server-side session read), but the hero still displayed "Sign in with your atproto account" — a flash of wrong UI before and after hydration.

Fix: converted `Home` to an `async` server component. Added `getSession()` call from `@/lib/session`. Branched the CTA region: authenticated users see a gradient "Browse profiles" button (→ `/browse`) and a "My profile" text link (→ `/profile/edit`); unauthenticated users see the original `<LoginButton />` and `<LoginErrorBanner />` unchanged. Server-side branch means no flash of the wrong state.

### 2. Mobile @handle Truncation

`src/components/Nav.tsx` had `max-w-[8rem] truncate` on the mobile handle span. At `text-xs`, 8 rem is roughly 19 characters — `@david-cornelson.bsky.social` (28 chars) was cut mid-word.

Fix: removed the fixed `max-w-[8rem]`. Restructured the flex layout of the mobile right-side row:
- Outer nav row: added `gap-3`
- Logo: `flex-shrink-0` so it never collapses
- Mobile right-side div: `min-w-0 flex-1 justify-end` so it shrinks gracefully and right-aligns
- About link: `flex-shrink-0` (priority sibling — always visible)
- Handle span: `truncate min-w-0` with no fixed max-width

The handle now displays in full when space allows; truncation is a graceful last-resort on genuinely narrow viewports, and About link is preserved because it has flex priority.

### 3. Removed Home Hero h1

The large `<h1>` displaying "Date" + gradient "Sky" span was removed from the home page. The Nav logo already brands the page; the standalone title was redundant. The hero now opens directly on the tagline.

## Key Decisions

### 1. Server component for auth branch on home page

A client-side auth check would have caused a flash-of-login-button before hydration. Reading the session server-side eliminates that flicker entirely and is consistent with how other auth-aware pages in the project work (e.g., `/browse`, `/profile/edit`).

### 2. flex-shrink / flex-1 over a fixed max-width on the handle

The prior `max-w-[8rem]` was a blunt fix that broke longer handles. Flexbox constraint-based layout is the right mechanism: the About link has explicit non-shrink priority, and the handle fills remaining space, truncating only when it must. This also survives short handles (no excess gap) and long handles (graceful truncation) without special-casing either.
