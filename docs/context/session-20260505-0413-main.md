### 3. Match Mode — Design Conversations and ADR

Spent significant session time on design before writing any implementation code. The conversation converged through several iterated proposals:

**Privacy split.** Match preferences (gender preference, age ranges, location filter, intent, toggle) are stored only in DateSky's private SQLite database and never published to the AT Protocol network. The user's own profile attributes (gender, age, location, tags, intentions) remain public as before.

**Tag cap rejected.** Initial design proposed a hard cap on tags. The user rejected this on UX grounds — it limits legitimate expression. Replaced with three layered defenses: opposition detection at save, length normalization in rank, and a minimum-overlap floor.

**Opposition registry proposed and accepted.** Six semantic axes where internal contradiction is unambiguous (diet, smoking, drinking, kids, relationship style, schedule). Detection fires at save time and produces a clear error naming the conflicting pair.

**Length normalization.** User proposed `tag_overlap / sqrt(candidate_tag_count)` — a standard document-length normalization (IR technique). Composes cleanly with the opposition check. Accepted.

**"Close" UX.** Two options were presented. User chose option B: per-axis breakdown ("8 outside age range · 3 different location · 1 different gender preference") over a single count. More actionable — viewer can see which filter to relax.

**Intent pool mapping.** `dating` pool = `dating` ∪ `casual` ∪ `long-term` (all flavors of romantic intent). `friends` pool = `friends` (plus `friendship` defensively). A candidate with only `casual` still appears for a dating-pool search.

All six locked decisions were recorded in **ADR 0001** (`docs/adrs/0001-match-mode.md`).

## Key Decisions

### 1. Match preferences are private, not published to AT Protocol

Preferences describe the viewer's desires about candidates and must not become observable attributes of the viewer. Exposing them to the network would allow inferring dating intent, gender preferences, and age range of interest — all of which are more sensitive than the profile attributes they operate on. Private SQLite storage is the only acceptable home for this data. Any future cross-app interop would require a new ADR explicitly overriding this.

See ADR 0001, Decision §1.

### 2. No tag cap — opposition detection + length normalization instead

Capping tags limits expression; the long tail of niche interests is exactly what a dating platform wants to enable. The chosen alternative is a composed set of defenses: opposition detection rejects internally contradictory profiles at save, length normalization (`score = overlap / sqrt(tag_count)`) automatically penalizes shotgun-tagging at rank time, and the minimum-overlap floor (2 shared tags) drops semantically meaningless candidates before ranking begins.

See ADR 0001, Decision §4.

### 3. "Close" is a per-axis breakdown, not a count

Showing "12 near-matches" gives no actionable signal. Showing "8 outside age range · 3 different location · 1 different gender preference" tells the viewer exactly which filter is the friction point. This also avoids a UX anti-pattern where the viewer widens every preference simultaneously to capture the close matches.

See ADR 0001, Decision §3.

### 4. Pure function for the match algorithm

The ranking and filtering logic lives in `src/lib/match/index.ts` with no database or HTTP imports. This makes it possible to test the algorithm in isolation and keeps the browse API handler thin. The DB query and the algorithm are separate concerns and must stay that way.

### 5. Footer affiliation disclaimer as a global mount

The disclaimer is mounted once in `layout.tsx` rather than duplicated per page. Every page on the site — including future pages that do not yet exist — inherits it automatically. This is the correct default; opting out would require an explicit layout override with justification.

## Architectural Notes

### Server-side photo enforcement

Photos are stripped from the browse API response before transmission rather than hidden client-side. This matters because client-side hiding can be trivially bypassed with browser dev tools. The profile data that reaches the browser is already photo-free when `show_photos` is false. `<ProfileView>` does not need to know about the preference — it simply renders what it receives.

### Intent pool design

The AT Protocol lexicon defines four intention tokens: `dating`, `friends`, `casual`, `long-term`. For match mode purposes, `casual` and `long-term` are flavors of romantic intent and therefore belong in the dating pool alongside `dating`. This means a viewer seeking "dating" will see candidates who list only `casual` or only `long-term`, which is the correct behavior. The friendship pool is narrow: only `friends` (and the defensive alias `friendship`) qualifies. This mapping is encoded in the pure match function and must not be scattered across the codebase.

### No test suite

The project currently has no automated test harness. The CLAUDE.md rule requiring tests before committing is acknowledged but moot for now. Adding a Vitest harness is the correct next step and warrants an ADR before implementation — the choice of test runner, fixture strategy, and what to cover first will constrain multiple sessions.
