## Key Decisions

### 1. Full ES256 Keypair Rotation (not a kid relabel)

The plan listed this as option (b); the operator confirmed the choice. Rationale: not a security requirement (under hard cutover the old `client_id` is unreachable, so the old kid is unreferenceable — security gain from rotation is zero) but operator preference for clean separation between old and new identities. The full rotation costs one extra step at deploy time (capturing the private JWK) and zero ongoing cost afterward.

### 2. Private Key Kept Out of LLM Context

`scripts/generate-keys.ts` routes the private JWK exclusively to `process.stdout.write`; the session ran the script with stdout redirected to `/tmp/nomare-private-key.json`. The private key bytes never appeared in any tool result and never entered an API turn's input. Operator reads the file independently via their own terminal. This pattern should be followed for any future key rotation work.

### 3. Real-Path Test as Operator Runbook

Per CLAUDE.md rule 12a, OAuth integration requires a real-path test against the production code path. End-to-end automation is not feasible: the AT Protocol consent screen requires human interaction with the user's PDS. Rather than a stub script, verification was captured as a structured operator-driven runbook (`docs/runbooks/phase3-oauth-cutover.md`) — the same pattern as Phase 2's `scripts/test-dual-publish.ts` (operator-runs-it) but with stricter human-driven steps. The test executes against live `nomare.net` after Phase 4 brings the vhost up.

### 4. Cookie Rename Bundled with OAuth Cutover

Phase 1 deferred the `cookieName` change to avoid a second forced re-login. Bundling it in Phase 3 produces a single re-login event coincident with the `client_id` change. Users are prompted once; the UX cost is minimized. Pre-deploy comms language is specified in the runbook.
