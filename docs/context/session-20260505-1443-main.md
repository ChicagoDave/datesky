### Arc 2: Match-mode review and fix

After the login fix landed (commit `c3eb901`), the user reported that match mode was producing no results. Before touching code, the production SQLite DB (`data/datesky.db`) was queried directly to inspect `user_preferences` rows, the candidate pool, distinct gender values, and tag overlap. The review surfaced 5 issues; the user chose to fix them all.

#### Issues found

1. **Gender filter was substring-matching against heterogeneous self-id strings.** The index contains values like `Male`, `male`, `M`, `Female`, `female`, `Femme`, `Trans woman`, `Cis-hetero man`, `NB`, `enby`, `Non-Binary`, `trans femme`, `Non binary femme`. With the original `lower.includes(p.toLowerCase())` approach, picking "woman" did not match "female" (no `woman` substring), and picking "man" did not match "male" (no `man` substring). The most common gender preference produced near-zero matches.

2. **Tag floor was asymmetric.** Original code suspended the floor when the viewer had < 2 tags, but still required candidates to share ≥ 2 tags with the viewer. With the current index (75 profiles, most with 0 tags, only ~5 with ≥ 2), this killed almost every match — David himself has 12 tags but zero other profiles share 2+ tags with him.

3. **Null gender treated as auto-fail** while null age was treated as auto-pass. Inconsistent — partial profiles silently excluded from any gender-filtered match.

4. **David's own `match_mode_enabled` was 0 in prod** (config issue, not code). User notified.

5. **Empty-state was a dead-end.** When all candidates failed 2+ axes, the existing "close" breakdown stayed empty and the user only saw "No profiles match" with a generic widen-your-filters hint. No actionable signal about which filter killed the result.

#### Changes to `src/lib/match/index.ts`
- Added `GENDER_SYNONYMS` table mapping canonical preferences (woman/man/nonbinary/trans/genderqueer) to synonym sets:
  - `woman` accepts: woman, women, female, f, femme, womxn, lady, gal
  - `man` accepts: man, men, male, m, guy, dude
  - `nonbinary` accepts: nonbinary, non binary, nb, enby, gnc, agender
  - `trans` accepts: trans, transgender, transmasc, transfem, ftm, mtf, and others

- Added `normalizeGender()` (lowercase + non-alphanumeric → space + collapse whitespace) and `genderPreferenceMatches()` (whole-word match for single-token synonyms via Set lookup; substring match for multi-word synonyms like "non binary").
- Custom user-typed preferences not in the synonym table fall back to normalized substring match.
- `passesGender()` now returns `true` when candidate gender is `null` or normalizes to empty (consistent with `passesAge`).
- Tag floor now only enforced when both viewer.tags ≥ 2 AND candidate.tags ≥ 2. Sparse candidates pass the floor and rank with score 0 (bottom of list).
- Extended `MatchResult` with `pool` (input candidate count), `failsBy` (per-axis fail count, multi-counted across multi-axis failures), and `failedTagFloor`.

## Key Decisions

### 1. No auto-rewriting of domain guesses (Arc 1)
`normalizeHandle()` strips `@` (unambiguous) but does not auto-append `.bsky.social` or rewrite `.bsky.app`. Those are guesses about PDS membership that could silently route to the wrong host. Better to surface a clear error and let the user correct it.

### 2. 400 vs 500 for resolution errors (Arc 1)
Identity resolution failures are caused by bad user input, not server faults. Returning `400` is semantically correct and prevents generic "Server Error" noise in monitoring.

### 3. Symmetric tag floor (Arc 2)
The floor (≥ 2 shared tags) only fires when both sides have ≥ 2 tags. A sparse candidate (0–1 tags) is not penalized — they simply rank at score 0. This keeps the index useful during early growth when most profiles have few tags.

### 4. Synonym table over runtime normalization (Arc 2)
Gender self-identification strings are culturally specific and drift over time. A static synonym table with a defined canonical set is easier to audit, extend, and test than a fuzzy-match or embedding approach. The fallback (normalized substring match) handles unanticipated tokens without hard-failing.

## Root Cause (Arc 1 — log analysis)
All failures visible in `journalctl -u datesky -n 200` were `IdentityResolverError` from `client.authorize()` caused by malformed handles:
- Leading `@` (Twitter muscle memory): `@dashisnew.bsky.social`
- Bare username: `dashisnew`
- `@` as domain separator: `dkitch01@bsky.social`
- Wrong TLD: `ozruta.bsky.app`
- Valid format, nonexistent handle: `david.cornelson.bsky.social`

The spinner lockup was a separate but compounding issue: the client only checked `data.url`, never `res.ok`, and only cleared `loading` in the catch block — so a `500 { error }` response left the button frozen indefinitely.
