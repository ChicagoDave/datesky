Aviary — Mobile Redesign

New name, new posture. Aviary is identity-neutral as a metaphor — many distinct beings coexisting under one roof — and the visual system reflects that: the default palette is warm dusk, not a flag, so a straight cis user doesn't read the homepage as "not for me." Identity-flag theming is an opt-in personalization, not a forced visual statement. Trifecta (dating + friends + community) is first-class throughout.

Each mode is independently togglable with its own preferences. A user open to friends and community but not dating just toggles Dating off — the app respects that and doesn't surface dating-specific affordances.

Design notes

1. Why the default brand isn't a flag anymore

The previous mockup used the bisexual-flag gradient as the default wordmark for everyone — even users who hadn't signed up. That double-purposes the gradient: it's both the brand's neutral state and its queer-affirming signal at the same time. A straight cis person landing on the homepage reads "this isn't for me" before they've even read the copy.

The fix is to decouple the default from the identity-affirming layer. Default is warm-dusk (coral + lavender + sage) — gallery-magazine palette, gender/orientation-neutral, and crucially still pretty. Adaptive theming becomes opt-in personalization. Identity captured on the profile drives matching, not visual theme.

2. The cross-theme rule

  • Your shell uses your theme — top chrome, segmented nav, your filter pills, your insight rows.
  • A profile detail uses the viewed person's theme inside their hero region — so visiting Maya's profile literally tints to her identity. You experience the other person's theme as you read their profile.
  • Action buttons stay in your theme — Connect / Save are about your intent, so the affordance stays consistent.
  • Browse cards get a thin top accent line in the viewed person's theme — visually heterogeneous list rather than a uniform Tinder grid.

3. Why opt-in theming respects more users than auto-themed-by-identity

Some queer users — closeted, in hostile contexts, or just preferring quiet — don't want a flag-themed app. Some straight users want personalization (just not a flag). Auto-applying a theme based on identity forces a public visual statement on everyone. Letting users pick separately respects autonomy. A bi user can pick Trans-flag because a friend who came out got them thinking. A cis straight user can pick Forest. A queer person in a hostile workplace can stay on Dusk. All three are first-class users.

4. Trifecta as a first-class peer model

Dating, Friends, and Community are independent modes — not a hierarchy. The browse mode-pills filter the list by which modes a person is open to. Each card surfaces the modes that person is open to as small chips, alongside the tier badge. The profile-edit "Looking for" section is three independent toggleable cards, each with its own preferences (age range and genders for Dating; tag-overlap floor for Friends; tier and frequency for Community).

5. What the bird mark does

The wing SVG in the wordmark uses the same three theme variables (--c1, --c2, --c3) as the wordmark's gradient. Every theme paints its own bird without us having to ship a different SVG. The metaphor and the technical implementation are one move.

6. What's the same as the prior mockup

  • Top sticky header replaces the iOS Safari bottom-tab wobble — unchanged decision
  • Community / Extended / Distant tiers from ZIP — unchanged
  • Tag-overlap insight row — unchanged but now per-mode-aware ("3 open to dating, 4 to friends, 5 to community")
  • Tier badge in profile detail hero — unchanged

7. What's deferred

  • Trademark check on "Aviary" (Adobe owned a defunct photo-editor brand under this name)
  • How a user's theme actually ends up in the database (single token? per-region overrides?)
  • Photo support on cards — still text-only
  • Onboarding flow — how a new user picks identity / theme / intentions
  • International ZIP/postcode adjacency
  • Match-mode reconciliation — does the existing scoring fold into "Community + 4+ shared tags" as a default Dating sort?
Open questions for the next pass:
  • Do we surface the cross-theme on browse cards (thin accent line per card), or only on profile detail? Cards lean noisy if every one has its own color line; restraint may be better.
  • Should the action buttons on a profile detail use the user's theme (current proposal) or the viewed person's theme (more immersive but breaks affordance consistency)?
  • Privacy floor for tier counts — when "Community" has only 1 person, do we hide the count to protect them? "< 5" → "a few"?
  • Is "Community" mode meaningfully different from "Friends" mode for matching, or do they collapse to the same thing in practice? The distinction the trifecta wants to draw is "individual relationship" vs "place-rooted recurring connection" — does the data model support that?
  • Onboarding — who picks intentions / identity / theme first, and in what order? Trifecta suggests intentions are the foundational sort, but ZIP / community is the geographic root.
