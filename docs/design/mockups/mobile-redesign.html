Design notes

1. Why the bottom tab bar is gone

iOS Safari's URL bar slides in/out as the user scrolls. A position: fixed element anchored to the bottom tracks the visual viewport, which produces the "wobble" — and during the animation, hit-testing intermittently misses the tab cell, which is why taps don't always land. Backdrop-blur on a fixed element makes the wobble visible (the blur layer recomposites every frame).

The fix isn't to tweak padding — the pattern is wrong for iOS. Putting nav at the top (above the URL-bar zone) sidesteps all of it. The segmented control inside a sticky header gives us three tap targets ≥ 44px, no wobble, no compositor cost, and it's symmetrical with the existing desktop nav.

2. Communities by ZIP — the model

  • Community = same ZIP code as you
  • Extended = an adjacent ZIP (lookup table, computed once per ZIP)
  • Distant = everyone else

ZIP is stored privately. Only the derived tier is ever shown to other users. The public location field stays a free-text neighborhood string ("The Loop", "Logan Square") so users control what's visible. We need a ZIP-adjacency dataset; for the US, the Census Bureau publishes ZCTA neighbor lists, or we can derive adjacency from ZIP centroid distance < ~5 mi as a simpler fallback.

3. What replaces free-text "location" search

The current browse has a free-text location filter that's basically useless without geocoding. Tier strip replaces it: pick Community / Extended / Distant, then filter by tag and intention within that tier. Distance is no longer a slider — it's three buckets that map to a real social experience ("could meet for coffee tomorrow" vs "worth the train ride" vs "long-distance").

4. Tag-overlap insight row

Borrowed from thefold's "latent demand" idea. Before showing the cards, we tell you the headline: "5 in your community share 4+ of your tags." That's the answer to "should I bother browsing this tier?" and it's the kind of thing that makes someone open the app daily.

5. Card tier badges + shared-tag count

Every card now leads with the tier badge (Community / Extended / Distant) and a shared-tag count. This is why the user is seeing this person — surfacing it is more honest than a sorted list with no labels.

6. What's deferred

  • International ZIPs (Canadian postal codes, UK postcodes) — the model generalizes but the adjacency dataset is per-country
  • Match-mode integration — match-mode's existing scoring needs to factor in tier or get rolled into the tier UI
  • Photos — current cards are text-only; photo support is a separate design pass
  • Notification / messaging surfaces — out of scope for this redesign
Open questions for the next pass:
  • Do we keep "match mode" as a separate concept, or fold it into "Community + 4+ shared tags" as a default sort?
  • What about users with rural ZIPs that have zero adjacent populated ZIPs? Do we widen to a centroid-radius for them?
  • Should the tier strip persist on the profile-detail screen for context, or is the badge in the hero enough?
  • Is there a privacy concern with surfacing exact ZIP-overlap counts when communities are small (e.g., "1 in your community" reveals more than intended)? Probably need a floor — "< 5" → "a few".
